# '[[시맨틱 CSS]]'와 '[[유틸리티 우선]]'의 철학 전쟁
[[250918-Tailwind|Tailwind CSS]]가 처음 등장했을 때, [[250918-Front-end|프론트엔드]] 커뮤니티는 거대한 '철학 전쟁'에 휩싸였습니다.

- **전통주의자: '시맨틱(Semantic) CSS' 진영**
    - **주장:** "[[250918-HTML|HTML]]은 문서의 '의미(semantic)'를, [[250918-CSS|CSS]]는 '보여지는 방식(presentation)'을 담당해야 한다. 이 둘은 철저히 분리되어야 한다!"
    - **좋은 예 ([[250918-BEM|BEM 방법론]]):**
```HTML
<div class="user-profile">
	<img class="user-profile__avatar" src="..." />
	<p class="user-profile__name">Solmin</p>
</div>
```
        
HTML만 봐도 구조와 의미를 알 수 있고, 스타일은 `.css` 파일 안에서 `.user-profile__avatar` 같은 클래스에 정의됩니다.
        
- **혁명가: '유틸리티 우선(Utility-First)' 진영 (Tailwind)**
    - **주장:** "어차피 컴포넌트 시대에 그 분리는 허상이다! 스타일은 구조와 뗄 수 없는 관계다. CSS 파일을 왔다 갔다 하며 컨텍스트를 전환하는 비용이 더 크다!"
    - **Tailwind 방식:**
```HTML
<div class="p-4 border rounded-lg flex items-center">
	<img class="w-16 h-16 rounded-full mr-4" src="..." />
	<p class="text-lg font-bold">Solmin</p>
</div>
```
        
- **초기의 비판:** "이건 20년 전 `style="..."` 인라인 스타일로 회귀하는 끔찍한 방식이다! HTML이 더러워진다!"
        

**왜 Tailwind가 승리하고 있는가?**
이 논쟁에서 Tailwind가 수많은 개발자들의 마음을 얻은 이유는, **리액트(React), Vue, Svelte 같은 '컴포넌트 기반 프레임워크'의 시대**가 열렸기 때문입니다.

더러워 보이는 유틸리티 클래스 덩어리는, 결국 `<UserProfile />` 이라는 **하나의 '컴포넌트' 안에 [[250918-Encapsulation|캡슐화(encapsulation)]]** 됩니다. 컴포넌트를 사용하는 사람은 그 복잡한 내부를 볼 필요가 없죠.

결과적으로 개발자는 `.css` 파일을 거의 열지 않고, 오직 컴포넌트 파일 안에서 구조(JSX), 로직(JS), 스타일(Tailwind)을 한번에 해결할 수 있게 되었습니다. 이는 엄청난 개발 속도 향상과 컨텍스트 전환 비용 감소로 이어졌고, 이것이 바로 Tailwind가 현대 프론트엔드 개발의 대세 중 하나로 자리 잡게 된 이유랍니다!
